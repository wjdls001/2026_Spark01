## 0. Figma 와이어프레임 반영 요약

Figma Page 1은 다음 6개 섹션으로 구성된다.

1. 로그인/사용자 조사 온보딩
2. 홈
3. 챌린지
4. 번개 모임
5. 마이페이지
6. 운동시작

구현 시 기존 요구사항보다 Figma 화면 플로우를 우선한다. 특히 다음 사항을 반드시 반영한다.

- 로그인 버튼은 Apple / Google / Kakao 순서다. Naver 버튼은 Figma에 없으므로 제외한다.
- 온보딩은 약관 동의 후 전화번호 인증을 거쳐 사용자 조사 7단계로 진행한다.
- 사용자 조사 7단계는 `기본 정보 → 운동 목표 → 운동 성향 → 좋아하는 운동 → 자주 하는 운동 → 운동 빈도 → 평균 운동 시간` 순서다.
- 번개 모임 탭 기본 진입 화면은 리스트가 아니라 지도다.
- 운동 시작은 종목 선택과 목표 입력을 한 화면에서 처리한다.
- 같이 운동은 모임장/모임원 준비 화면과 운동 세션 화면을 분리한다.
- 마이페이지는 프로필, 운동 관리, 활동 관리 3탭 구조다.

## REQ-01. 인증 — 소셜 로그인

**우선순위:** P0  
**대상 화면:** `/login`  
**Figma 화면:** `소셜로그인-로그인 시작`

### 요구사항

- 로그인 시작 화면에는 서비스명/슬로건 영역과 소셜 로그인 버튼을 표시한다.
- 소셜 로그인 버튼 순서는 `Continue with Apple` → `Continue with Google` → `Continue with Kakao`로 고정한다.
- 기존 문서에 있던 네이버 로그인 버튼은 Figma에 없으므로 MVP 구현 대상에서 제외한다.
- MVP에서는 실제 OAuth가 준비되지 않은 경우 버튼 클릭 시 mock 로그인 또는 Supabase Auth 이메일 계정으로 대체 진입할 수 있다.
- 로그인 성공 후 온보딩 완료 여부를 확인한다.
- 약관 미동의 사용자는 `/onboarding/terms`로 이동한다.
- 전화번호 인증 미완료 사용자는 `/onboarding/phone`으로 이동한다.
- 사용자 조사 미완료 사용자는 `/onboarding/profile`로 이동한다.
- 모든 온보딩을 완료한 사용자는 `/home`으로 이동한다.

### 완료 기준

- [ ] Apple / Google / Kakao 버튼이 순서대로 표시된다.
- [ ] 로그인 성공 후 온보딩 진행 상태에 따라 올바른 화면으로 분기된다.
- [ ] 미인증 사용자가 보호 경로에 접근하면 `/login`으로 리다이렉트된다.

## REQ-02. 온보딩 — 약관 동의, 전화번호 인증, 사용자 조사 7단계

**우선순위:** P0  
**대상 화면:** `/onboarding/terms`, `/onboarding/phone`, `/onboarding/profile`, `/onboarding/result`  
**Figma 화면:** `소셜로그인-약관 동의 및 프로필 확인`, `소셜로그인-전화번호 인증`, `사용자조사-*`, `운동 레벨 결과`

### 요구사항

- 소셜 로그인 후 약관 동의 화면으로 이동한다.
- 약관 동의 항목은 다음과 같다.
  - 전체 동의하기
  - 이용약관 동의(필수)
  - 개인정보 수집 및 이용 동의(필수)
  - 마케팅 정보 수신 동의(선택)
- 필수 약관 미동의 시 다음 단계로 이동할 수 없다.
- 약관 상세 보기 클릭 시 이용약관/개인정보 처리방침 스크롤 화면 또는 모달을 제공한다.
- 약관 동의 후 전화번호 인증 화면으로 이동한다.
- 전화번호 인증은 MVP에서 mock 인증번호로 처리 가능하되, 화면과 상태값은 구현한다.
- 전화번호 인증 후 사용자 조사 7단계를 진행한다.

### 사용자 조사 7단계

| Step | Figma 화면명                  | 입력 항목                           | 저장 대상                                                                      |
| ---- | ----------------------------- | ----------------------------------- | ------------------------------------------------------------------------------ |
| 1    | 사용자조사-기본 정보 입력     | 닉네임, 나이, 성별                  | `profiles.nickname`, `profiles.birth_year` 또는 `age_range`, `profiles.gender` |
| 2    | 사용자조사-운동목표           | 운동 목표                           | `profiles.exercise_goal` 또는 onboarding 응답 필드                             |
| 3    | 사용자조사-운동 성향 조사     | 운동 성향 태그 복수 선택, 직접 입력 | `profiles.workout_traits`                                                      |
| 4    | 사용자조사-좋아하는 운동 선택 | 좋아하는 운동 복수 선택, 직접 입력  | `profiles.preferred_sports`                                                    |
| 5    | 사용자조사-자주하는 운동 선택 | 자주 하는 운동 1개 이상             | `profiles.frequent_sports` 또는 onboarding 응답 필드                           |
| 6    | 사용자조사-운동 빈도 조사     | 주간 운동 빈도                      | `profiles.weekly_exercise_frequency` 또는 onboarding 응답 필드                 |
| 7    | 사용자조사-평균 운동 시간     | 회당 평균 운동 시간                 | `profiles.average_workout_duration_range` 또는 onboarding 응답 필드            |

#### 와이어프레임 확정 입력값 및 인터랙션 (2026-06-18)

- 공통: 흰 배경, 상단 헤더, `Step n of 7`, 단계별 퍼센트 `10/24/38/54/68/82/99`를 유지한다. 디자인 시스템에 따라 진행/선택은 Purple, 주요 하단 CTA는 Lime을 사용한다.
- 기본 정보: 닉네임 2~10자와 실제 DB 중복 확인, 연령대 `10대/20대/30대/40대/50대/60대 이상`, 성별 `남성/여성/이외`.
- 닉네임 확인: 사용 가능과 중복 상태를 각각 팝업으로 표시한다.
- 성별 안내: `이외` 선택 시 맞춤 운동 모임 추천 정확도 안내 팝업을 표시한다.
- 운동 목표: `다이어트/벌크업/체력 증진·건강 유지` 단일 선택.
- 운동 성향: 지정 태그 복수 선택과 10자 이내 직접 입력.
- 좋아하는 운동: 1개 이상 복수 선택, 선택 순서 배지, 10자 이내 직접 입력.
- 자주 하는 운동: `러닝/사이클링/웨이트/홈 트레이닝` 또는 직접 입력 단일 선택.
- 운동 빈도: `거의 안 함/주 1~2회/주 3~4회/주 5회 이상` 단일 선택.
- 평균 시간: `30분 미만/30~60분/60~90분/90분 초과` 단일 선택.
- 결과: 응답으로 산정한 레벨, 운동 취향, 빈도를 요약하고 홈 진입 CTA를 제공한다.
- 실제 DB에 전용 필드가 추가되기 전까지 추가 응답은 `profiles.workout_traits`에 `연령대:`, `운동목표:`, `자주하는운동:`, `주간빈도:`, `평균운동시간:`, `선호운동:` 접두어로 직렬화한다.
- 로그인 후 단순히 `profiles` 행 존재 여부만 확인하지 않는다. `운동목표:`, `자주하는운동:`, `주간빈도:`, `평균운동시간:` 마커가 모두 있을 때만 사용자 조사 완료로 판단하며, 하나라도 없으면 `/onboarding/terms`부터 온보딩을 시작한다.
- 온보딩 저장은 사용자 ID 기준 upsert로 처리하여 기존 미완료 프로필도 조사 결과로 정상 갱신한다.

- Step 3과 Step 4에는 직접 입력 팝업을 제공한다.
- 직접 입력 팝업은 운동명 또는 성향명을 10자 이내로 입력받고 `추가하기` 버튼으로 선택 목록에 반영한다.
- 사용자 조사 완료 후 운동 레벨 결과 화면을 표시한다.
- 결과 화면에는 `{등급명} · LV.{숫자}`(예: `비기너 · LV.8`), `SPARK 이시네요!`, 프로필 요약, 운동 초보자/중급자/숙련자 등 레벨 결과를 표시한다. 등급명/숫자 매핑은 CLAUDE.md 0.2.1·0.2.8과 `src/lib/utils/level.ts` 기준을 따른다.
- 결과 화면의 완료 CTA 클릭 시 `/home`으로 이동한다.

### 완료 기준

- [ ] 약관 동의 → 전화번호 인증 → 사용자 조사 7단계 → 결과 → 홈 순서로 이동한다.
- [ ] 7단계 진행률이 Figma처럼 10%, 24%, 38%, 54%, 68%, 82%, 99%로 표시된다.
- [ ] 닉네임 중복 확인이 동작한다.
- [ ] 직접 입력 팝업에서 입력값이 선택 목록에 추가된다.
- [ ] 온보딩 완료 후 프로필 데이터가 저장된다.

## REQ-03. 홈 — 로그인 후 대시보드와 권한 팝업

**우선순위:** P0  
**대상 화면:** `/home`, `/notifications`  
**Figma 화면:** `홈 - 로그인 후 - 화면`, `홈-권한 승인 팝업-처음 로그인 했을때만 표시`, `알람창-전체보기`, `알람창-읽지않음`, `홈 - 운동 상세보기 팝업`, `홈 - 번개 상세 조회 탭`

### 요구사항

- 홈 상단에는 `OO님 운동할 준비가 되셨나요?` 인사말과 이번 주 예정 모임 수를 표시한다.
- 운동 레벨 뱃지, 연속 운동 일수, 총 운동 횟수, 총 칼로리, 총 거리 등 요약 지표를 카드로 표시한다.
- 최근 운동 기록 카드를 클릭하면 운동 상세 팝업 또는 `/mypage/workout/:sessionId`로 이동한다.
- 추천/예정 번개 카드를 클릭하면 번개 상세 화면으로 이동한다.
- 최초 로그인 후 한 번만 권한 승인 팝업을 표시한다.
- 권한 승인 팝업 항목은 푸시 알림, 건강 데이터 접근, 위치 데이터 접근이다.
- 알림 아이콘 클릭 시 `/notifications`로 이동하거나 알림 패널을 열고, 전체/읽지 않음 탭을 제공한다.
- 알림 화면에는 주요 알림, 번개 모임 알림, 삭제, 참가하기 액션을 제공한다.

### 완료 기준

- [ ] 홈 진입 시 Figma의 인사말/운동 요약 카드가 표시된다.
- [ ] 최초 로그인 시 권한 승인 팝업이 1회 표시된다.
- [ ] 알림 전체/읽지 않음 탭이 동작한다.
- [ ] 운동 기록 카드와 번개 카드의 상세 이동이 동작한다.

## REQ-04. 운동 시작 — 혼자 운동 플로우

**우선순위:** P0  
**대상 화면:** `/exercise`, `/exercise/solo/setup`, `/exercise/solo/routine`, `/exercise/solo/session`, `/exercise/result`  
**Figma 화면:** `혼자 운동-종목 선택+목표 입력`, `혼자 운동-루틴 불러오기`, `혼자 운동 화면-시작 전`, `혼자 운동 화면-시작 후`, `운동 시작 전 목표 수정`, `혼자 운동-선택 종목 종료 후`, `혼자 운동 종료 후 화면`

### 요구사항

- 운동 탭 진입 시 혼자 운동과 같이 운동 진입을 제공한다.
- 혼자 운동은 종목 선택과 목표 입력을 한 화면에서 처리한다.
- 종목군별 UI를 분기한다.
  - 걷기/러닝/자전거: 거리, 시간, 경로/지도 영역, 목표 수정
  - 헬스/홈트: 루틴, 세트, 시간 중심 입력
  - 종목 직접 입력: 직접 입력 팝업을 제공하고 입력 후 목표 입력 화면으로 합류
- 사용자는 저장된 루틴을 불러올 수 있다.
- 운동 시작 전 목표 수정 화면 또는 바텀시트를 제공한다.
- 세션 시작 전 화면과 시작 후 화면을 구분한다.
- 세션 화면에는 타이머 `00:00:00`, 운동 종목명, 지도 또는 운동 정보 영역을 표시한다.
- 운동 종료 후 기록 입력 화면을 표시하고, 최종 결과 화면에서 칭찬 문구와 보상/요약을 제공한다.
- 운동 종료 시 `exercise_sessions`에 mode=`solo`로 저장한다.

### 완료 기준

- [ ] 종목 선택+목표 입력 화면이 Figma처럼 한 화면으로 구현된다.
- [ ] 걷기/러닝/자전거와 헬스/홈트 UI가 분기된다.
- [ ] 종목 직접 입력 팝업이 동작한다.
- [ ] 루틴 불러오기 화면이 연결된다.
- [ ] 시작 전/시작 후/종료 후 화면이 순서대로 연결된다.
- [ ] 운동 기록이 `exercise_sessions`에 저장된다.

## REQ-05. 운동 시작 — 같이 운동 플로우

**우선순위:** P0  
**대상 화면:** `/exercise/spark/select`, `/exercise/spark/host-ready/:sparkId`, `/exercise/spark/member-ready/:sparkId`, `/exercise/spark/session/:sparkId`, `/exercise/result`  
**Figma 화면:** `같이 운동-소속 모임 있음, 모임장 화면`, `같이 운동-소속 모임 있음, 모임원 화면`, `모임장 운동세션 화면`, `모임원 운동세션 화면`

### 요구사항

- 같이 운동 선택 시 승인 완료된 참여 번개 또는 개설 번개가 있는지 확인한다.
- 승인된 번개가 없으면 번개 탐색(`/sparks`) 또는 번개 생성(`/sparks/new`)으로 이동할 수 있게 한다.
- 승인된 번개가 있으면 번개 선택 목록을 표시한다.
- 선택한 번개에서 현재 사용자의 role이 `host`면 모임장 준비 화면으로 이동한다.
- 현재 사용자의 role이 `member`면 모임원 준비 화면으로 이동한다.
- 모임장 준비 화면에는 미접속 멤버 알림 보내기, 멤버 접속 상태, 운동 시작 버튼을 제공한다.
- 모임원 준비 화면에는 모임 정보, 접속 상태, 운동 시작 또는 대기 상태를 제공한다.
- 운동 세션 종료 시 `exercise_sessions`에 mode=`spark`, `spark_id`를 함께 저장한다.

- 건강 앱 연결 화면에는 Apple Health, Samsung Health 연동 UI를 제공한다.
- 고객센터/문의 화면에는 운영 시간과 1:1 문의하기를 제공한다.
- 1:1 문의 작성 화면에는 문의 유형, 제목, 내용을 입력받는다.
- 회원탈퇴 화면에는 삭제/취소 안내와 주의사항을 표시한다.

### 완료 기준

- [ ] 설정 홈에서 모든 하위 설정으로 이동 가능하다.
- [ ] 기기/건강 앱 연결 UI가 표시된다.
- [ ] 1:1 문의 작성 화면이 동작한다.
- [ ] 회원탈퇴 주의사항 확인 UI가 있다.

## REQ-15. 공통 — 하단 내비게이션 및 모바일 레이아웃

**우선순위:** P0  
**대상:** 전체 인증 화면

### 요구사항

- 하단 탭 구성은 홈(`/home`), 운동(`/exercise`), 모임(`/sparks`), 챌린지(`/challenges`), 프로필(`/mypage`)이다.
- 현재 활성 탭은 색상과 아이콘 강조로 구분한다.
- 로그인, 약관, 전화번호 인증, 온보딩, 결과 화면에서는 하단 탭을 숨긴다.
- Figma 기준 운동 세션 화면에는 하단 탭 텍스트가 포함되어 있으므로 MVP에서는 세션 중에도 하단 탭을 유지한다.
- 단, 세션 중 다른 탭으로 이동하려 하면 `운동을 종료하지 않고 나가시겠어요?` 확인 모달을 띄운다.
- 기본 화면 폭은 390px이며, 데스크탑에서는 중앙 정렬 컨테이너로 표시한다.

### 완료 기준

- [ ] 5개 탭 내비게이션이 표시된다.
- [ ] 온보딩 계열 화면에서는 하단 탭이 숨겨진다.
- [ ] 세션 중 이탈 확인 모달이 표시된다.
- [ ] 390px 기준 레이아웃이 깨지지 않는다.

## REQ-16. 공통 — 로딩/에러/빈 상태 처리

**우선순위:** P0  
**대상:** 전체 데이터 조회 화면

### 요구사항

- 데이터를 불러오는 모든 화면에서 로딩 상태를 표시한다.
- 네트워크 오류 또는 서버 오류 발생 시 에러 메시지와 재시도 버튼을 표시한다.
- 데이터가 없는 경우 기능에 맞는 빈 상태 메시지와 행동 유도 버튼을 표시한다.
- 표준 빈 상태 메시지:
  - 번개 없음: `근처에 열린 번개가 아직 없어요. 직접 만들어볼까요?`
  - 운동 기록 없음: `첫 운동 기록을 만들어보세요.`
  - 챌린지 없음: `진행 중인 챌린지가 없어요.`
  - 참여한 번개 없음: `아직 참여한 번개가 없어요.`

### 완료 기준

- [ ] 주요 화면에 로딩 상태가 있다.
- [ ] 오류 발생 시 재시도 가능하다.
- [ ] 빈 상태별 CTA가 있다.

## REQ-17. 보안 — RLS 및 접근 제어

**우선순위:** P0  
**대상:** 전체 Supabase 데이터 접근

### 요구사항

- 모든 public 스키마 테이블에 RLS를 활성화한다.
- 클라이언트 코드에는 Supabase anon key만 사용한다.
- 사용자는 자신의 운동 기록, 알림, 챌린지 진행, 기기 연결, 약관 동의 데이터만 조회/수정할 수 있다.
- 번개 수정/삭제는 host_id가 현재 사용자인 경우에만 허용한다.
- 참여 승인/거절은 해당 번개의 host만 처리할 수 있다.
- 신고 내용은 신고자 본인과 관리자만 조회할 수 있다.

### 완료 기준

- [ ] 타인의 개인 데이터에 접근할 수 없다.
- [ ] 비호스트가 번개를 수정할 수 없다.
- [ ] service role key가 프론트엔드에 없다.

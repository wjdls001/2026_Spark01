## REQ-01. 인증 — 소셜 로그인 및 이메일 회원가입

**우선순위:** P0  
**대상 화면:** `/login`, `/signup`  
**와이어프레임 반영:** 소셜 로그인 우선 UI (카카오/Apple/네이버 버튼)

### 요구사항

- 로그인 화면 진입 시 소셜 로그인 버튼(카카오, Apple, 네이버)을 우선 표시한다.
- 이메일 로그인은 "이메일로 로그인" 버튼을 누른 후 폼을 표시하는 방식으로 분리한다.
- MVP에서 실제 소셜 OAuth 연동 전까지 소셜 버튼은 UI만 제공하며, 실제 동작은 이메일 방식으로 처리한다.
- 사용자는 이메일과 비밀번호로 회원가입할 수 있다.
- Supabase Auth를 통해 인증 처리하며, 클라이언트에는 anon key만 사용한다.
- 로그인하지 않은 사용자가 보호된 경로에 접근하면 `/login`으로 리다이렉트한다.
- 로그인 성공 후 온보딩 완료 여부를 확인하고, 미완료 시 온보딩 플로우로 이동한다.
- 로그인 성공 후 온보딩이 완료된 사용자는 `/home`으로 이동한다.

### 완료 기준

- [x] 소셜 로그인 UI (카카오/Apple/네이버) 표시
- [ ] Supabase Auth 이메일 회원가입 동작
- [ ] 로그인/로그아웃 정상 작동
- [ ] 미인증 접근 시 `/login` 리다이렉트
- [ ] 인증 상태가 새로고침 후에도 유지됨

## REQ-02. 온보딩 — 약관 동의 및 프로필 초기 설정

**우선순위:** P0  
**대상 화면:** `/onboarding/terms`, `/onboarding/profile`

### 요구사항

- 신규 회원은 회원가입 직후 약관 동의 화면으로 이동한다.
- 약관 동의 항목: 서비스 이용약관(필수), 개인정보 처리방침(필수), 위치정보 이용약관(선택), 마케팅 수신 동의(선택).
- 필수 약관에 동의하지 않으면 다음 단계로 진행할 수 없다.
- 약관 동의 이력은 `terms_agreements` 테이블에 저장한다.
- 약관 동의 완료 후 프로필 설정 화면으로 이동한다.
- 프로필 설정은 와이어프레임 기준 7단계로 구성한다.
  1. 닉네임 + 활동 지역
  2. 운동 목적 (건강 유지/다이어트/스트레스 해소/사회적 교류/대회)
  3. 나의 운동 성향 태그 (복수 선택)
  4. 즐기는 운동 종목 (복수 선택)
  5. 운동 레벨 선택
  6. 선호하는 운동 파트너 유형
  7. 주간 운동 빈도
- 닉네임은 중복 확인을 거쳐야 하며, 중복 시 즉시 안내 메시지를 표시한다.
- 온보딩 완료 후 SPARK 시작 완료 화면(환영 화면)을 보여준 뒤 `/home`으로 이동한다.
- 프로필 저장 완료 시 `profiles` 테이블에 데이터를 저장한다.

### 완료 기준

- [x] 7단계 온보딩 플로우 구현
- [ ] 필수 약관 미동의 시 다음 단계 진행 불가
- [ ] `terms_agreements` 저장 확인
- [ ] 닉네임 중복 확인 동작
- [x] 온보딩 완료 화면(환영) 구현
- [ ] `profiles` 생성 후 홈 진입 확인

## REQ-03. 홈 — 운동 시작 허브 대시보드

**우선순위:** P0  
**대상 화면:** `/home`

### 요구사항

- 홈 화면 최상단 또는 가장 눈에 띄는 영역에 `운동 시작하기` CTA 버튼을 배치한다.
- 로그인한 사용자의 닉네임과 인사말을 화면 상단에 표시한다.
- **활동 링 카드**: 이번 주 운동 횟수/칼로리/거리를 SVG 원형 링 3개로 시각화한다.
- **자주 하는 운동 루틴**: 선호 운동 아이콘을 가로 스크롤로 표시한다.
- 주변 추천 번개 모임을 2열 그리드로 최대 3개 표시한다. 4번째 슬롯은 "새 번개 만들기" 버튼이다.
- **운동 레벨/신뢰도 카드**: 다크 구역 상단에 두 개의 미니 카드로 표시한다.
- 진행 중인 챌린지의 달성률을 다크 구역에 요약 카드로 표시한다.
- 최근 운동 기록 1~3개를 다크 구역에 표시한다.
- 읽지 않은 알림이 있으면 알림 아이콘에 뱃지(숫자)를 표시하고, 클릭 시 드롭다운으로 알림 목록을 표시한다.
- Supabase 데이터가 없을 때는 `src/lib/mockData.ts` 더미 데이터를 사용한다.

### 완료 기준

- [ ] `운동 시작하기` 버튼이 화면 최상단에 가장 크게 노출됨

## REQ-04. 운동 시작 — 혼자 운동 플로우

**우선순위:** P0  
**대상 화면:** `/exercise`, `/exercise/setup`, `/exercise/session`, `/exercise/result`

### 요구사항

- 운동 시작 화면에서 `혼자 운동`과 `같이 운동` 중 선택할 수 있다.
- 혼자 운동 선택 시 종목 선택 → 목표 설정 → 운동 세션 순서로 진행한다.
- 종목은 `sports` 테이블에서 불러오며, 사용자가 하나를 선택한다.
- 목표 설정 항목: 목표 시간(필수), 목표 거리(선택), 목표 칼로리(선택).
- 운동 세션 화면에서 경과 시간을 타이머로 표시한다.
- 운동 종료 시 `exercise_sessions` 테이블에 기록을 저장한다. mode는 `solo`.
- 운동 결과 화면에서 운동 시간, 거리(입력한 경우), 칼로리(입력한 경우), 획득 XP를 표시한다.
- MVP에서는 GPS 트래킹 없이 타이머 기반으로 구현하며, 거리·칼로리는 직접 입력 또는 mock 처리한다.

### 완료 기준

- [ ] 종목 선택 → 목표 설정 → 타이머 세션 플로우 연결
- [ ] 운동 종료 후 `exercise_sessions` 저장 확인
- [ ] 결과 화면에서 운동 데이터 표시

## REQ-06. 번개 모임 — 탐색 (리스트 및 지도)

**우선순위:** P1  
**대상 화면:** `/sparks`, `/sparks/map`

- 번개 모임 탭 진입 시 기본은 리스트 보기이며, 지도 보기로 전환할 수 있다.
- 리스트에는 번개 제목, 운동 종목(emoji 포함), 장소명, 예정 시간, 현재 참여 인원/정원, 호스트 신뢰도를 표시한다.

- **지도 보기**: Leaflet + OpenStreetMap으로 실제 지도를 표시한다 (API 키 불필요).
  - 각 번개의 위도/경도에 네온 핀(⚡)을 표시한다.
  - 핀 클릭 시 하단에 해당 번개의 미리보기 카드가 나타난다.
  - "상세 보기" 버튼으로 번개 상세 페이지로 이동한다.
- 번개 상세 상단에 장소 위치를 지도로 표시한다 (zoom 15).
- Supabase 데이터가 없을 때는 `MOCK_SPARKS` 더미 데이터를 사용한다.
- 리스트가 비어있을 때 "근처에 열린 번개가 아직 없어요. 직접 만들어볼까요?" 메시지와 번개 생성 버튼을 표시한다.

## REQ-07. 번개 모임 — 상세 조회 및 참여 신청

**우선순위:** P1  
**대상 화면:** `/sparks/:sparkId`

### 요구사항

- 번개 상세 화면에서 제목, 운동 종목, 장소, 날짜/시간, 정원, 현재 참여 인원, 호스트 닉네임·운동 레벨·신뢰 점수, 참여 조건(레벨, 성별, 연령), 상세 설명을 표시한다.
- 참여 신청 버튼을 누르면 다음 조건을 순서대로 검증한다.
  1. 로그인 여부 확인 (미로그인 시 `/login` 이동)
  2. 모집 상태가 `recruiting`인지 확인
  3. 정원 초과 여부 확인
  4. 이미 신청 또는 참여한 번개인지 확인
  5. 운동 레벨 조건 충족 여부 확인
- 신청 성공 시 `spark_participants`에 status=`requested`로 저장하고, "신청이 완료되었어요. 호스트 승인을 기다려주세요." 메시지를 표시한다.
- 이미 신청한 번개인 경우 참여 신청 버튼 대신 "승인 대기중" 상태를 표시한다.
- 이미 승인된 번개인 경우 "참여 확정" 상태를 표시한다.

### 완료 기준

- 날짜/시간은 현재 시각 이후만 선택 가능하다.

## REQ-09. 번개 모임 — 호스트 운영 (승인/거절/마감/종료)

**우선순위:** P1  
**대상 화면:** `/sparks/:sparkId/manage`

### 요구사항

- 호스트는 `requested` 상태의 참여 신청을 승인 또는 거절할 수 있다.
- 승인 시 `spark_participants.status`를 `approved`로 변경한다.
- 거절 시 `spark_participants.status`를 `rejected`로 변경한다.
- 정원이 다 찼을 때 호스트는 모집 마감 처리를 할 수 있다. 상태를 `closed`로 변경한다.
- 호스트는 번개 종료 처리를 할 수 있다. 상태를 `completed`로 변경한다.
- 호스트만 관리 페이지에 접근할 수 있으며, 비호스트 접근 시 상세 페이지로 리다이렉트한다.

### 완료 기준

- [ ] 승인/거절 후 `spark_participants.status` 변경 확인
- [ ] 모집 마감 후 `sparks.status=closed` 변경 확인
- [ ] 번개 종료 후 `sparks.status=completed` 변경 확인
- [ ] 비호스트 접근 차단

## REQ-10. 챌린지 — 목록 및 달성률 표시

**우선순위:** P2  
**대상 화면:** `/challenges`, `/challenges/:challengeId`

- 상단에 "오늘도 챌린지 진행중!" 배너 카드를 표시한다 (그라데이션, 전체 달성률 표시).
- 챌린지 탭은 `전체`, `운동`, `모임`, `이벤트` 4개 탭으로 구분하여 표시한다 (각 탭에 emoji 아이콘).
- 각 탭에는 해당 `challenge_type`의 챌린지 카드를 표시한다.
- 챌린지 카드 구성: 트로피 아이콘 영역 + 타입 배지 + 제목 + 설명 + XP 배지 + 진행률 바.
- 완료된 챌린지는 트로피 아이콘 배경이 `#C8FF3E`로 변경되고 "완료 ✓" 배지를 표시한다.
- 이벤트 챌린지 클릭 시 하단 시트 모달로 상세 내용(보상, 일시, 장소)을 표시한다.
- Supabase 데이터가 없을 때는 `MOCK_CHALLENGES` 더미 데이터를 사용한다.

### 완료 기준

- [x] 상단 배너 카드 (진행중 챌린지 요약)
- [x] 4개 탭 전환 및 타입별 챌린지 표시

## REQ-16. 보안 — RLS 및 접근 제어

**우선순위:** P0  
**대상:** 전체 Supabase 데이터 접근

### 요구사항

- 모든 public 스키마 테이블에 RLS를 활성화한다.
- 클라이언트 코드에서 service role key를 사용하지 않는다. anon key만 사용한다.
- 사용자는 자신의 `exercise_sessions`, `notifications`, `challenge_progress`, `device_connections`, `terms_agreements` 데이터만 조회/수정할 수 있다.
- `sparks` 수정/삭제는 `host_id`가 현재 사용자인 경우에만 허용한다.
- `spark_participants` 승인/거절은 해당 spark의 `host_id`인 사용자만 처리할 수 있다.
- 신고(`reports`) 내용은 신고자 본인만 조회할 수 있다.

### 완료 기준

- [ ] 타인의 개인 데이터에 접근 불가 확인
- [ ] 비호스트의 번개 수정 불가 확인
